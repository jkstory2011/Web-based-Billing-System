# 고객·계약 관리 + 청구서 생성 MVP 설계

## 배경 및 범위

전체 청구/정산 시스템은 다음 5개 핵심 기능으로 구성된다:
1. 고객 및 계약 관리
2. 청구서(Invoice) 생성 및 발송
3. 수납 및 결제 연동 (PG)
4. 미수금 및 연체 관리 (Dunning)
5. 정산 대시보드 및 통계

이들을 한 번에 설계·구현하기엔 범위가 크므로, 이 문서는 **1번(고객·계약 관리)과 2번(청구서 생성/발송)만을 MVP 범위**로 다룬다. 3~5번은 이 MVP가 만드는 데이터 모델(Contract, Invoice) 위에서 다음 단계로 순차 구축한다.

**비목표 (이 스펙에서 명시적으로 제외)**
- PG 결제 연동, 온라인 결제 처리
- 미수금/연체 자동 알림, 재청구 로직
- 매출 통계, 수납률 등 대시보드
- 멀티테넌트 (여러 회사가 함께 쓰는 SaaS 구조) — 단일 회사 내부용으로 한정
- 청구서 정기 자동 배치 생성 (cron) — MVP는 관리자가 버튼으로 수동 생성

단, `Invoice.status`는 향후 결제 연동 시 `PAID` 등 상태를 추가할 수 있도록 여유 있게 설계한다.

## 사용자 및 권한

내부 관리자만 시스템에 접근하며(고객은 별도 포털), 역할은 3단계로 나눈다.

| 기능 | SALES(영업) | ACCOUNTING(경리) | ADMIN(관리자) |
|---|---|---|---|
| 고객/계약 등록·수정 | O | 조회만 | O |
| 건별청구(Adhoc) 입력 | O | O | O |
| 청구서 생성/발행/발송 | X | O | O |
| 사용자·권한 관리 | X | X | O |

고객은 포털에 별도 계정(`PortalUser`)으로 로그인해 본인 청구서 목록과 PDF를 조회한다. 고객 계정 관리(생성/비활성화)는 ADMIN/SALES 권한.

## 아키텍처

모노레포, 3개 앱 구성:

```
billing-system/
├── apps/
│   ├── api/          # NestJS — 관리자용 + 고객포털용 API를 하나의 서비스로 통합
│   ├── admin-web/     # React SPA — 영업/경리/관리자용
│   └── portal-web/    # React SPA — 고객 전용 포털
└── packages/
    └── shared/         # 공통 타입(DTO), 상수
```

- API는 단일 서비스로 두되, 관리자 라우트(`/admin/*`)와 고객 라우트(`/portal/*`)를 역할 기반 가드(Guard)로 분리한다. 완전히 별도 백엔드 서비스로 나누는 것은 이 단계의 운영 부담 대비 이점이 작다고 판단해 채택하지 않았다.
- 프론트엔드는 관리자용(`admin-web`)과 고객용(`portal-web`)을 물리적으로 분리한 별도의 두 SPA로 만든다 (보안 경계를 명확히 하고 독립 배포를 가능하게 하기 위한 사용자 선택).
- DB: PostgreSQL 단일 인스턴스, 단일 스키마.
- 인증: 관리자/고객 각각 별도의 JWT를 발급하고(역할·클레임 분리), refresh token은 httpOnly 쿠키로 관리.

## 데이터 모델

**고객/계약**
- `Customer` — 개인/기업 구분(`type`), 이름/사업자번호, 연락처, 이메일
- `Contract` — 고객 FK, 계약기간(시작/종료), 상태(활성/해지 등)
- `ContractRecurringItem` — 정액 청구 항목. Contract FK, 주기(월/분기 등), 금액, 적용 시작~종료
- `AdhocCharge` — 건별 청구. Contract FK, 설명, 금액, 발생일, 입력자

**청구서**
- `Invoice` — Contract FK, 청구기간, 발행일/납부기한, 상태(초안/발행/발송완료), 합계금액
- `InvoiceLineItem` — Invoice FK, 설명, 수량/단가/금액, 출처 구분(`recurring`|`adhoc`) 및 원본 레코드 참조
- `InvoicePdf` — 생성된 PDF 파일 경로/버전 (재발행 시 이력 보존을 위해 버전 관리)

**사용자**
- `AdminUser` — 이메일/비밀번호(해시), `role`(SALES|ACCOUNTING|ADMIN)
- `PortalUser` — Customer와 1:1, 이메일/비밀번호(해시) — 고객 포털 로그인 전용, AdminUser와 별도 테이블

**핵심 관계**: 고객(1) — 계약(N) — [정액항목(N) + 건별청구(N)]. 청구서 생성 시점에 해당 계약의 미청구 정액항목/건별청구를 모아 `Invoice` + `InvoiceLineItem`으로 스냅샷한다 (원본 항목이 나중에 수정되어도 이미 발행된 청구서 금액은 변하지 않음).

## 핵심 흐름: 청구서 생성 (경리, 수동 트리거)

1. 경리가 청구 대상 기간(예: 2026-07)을 선택
2. 시스템이 활성 계약 중 해당 기간에 미청구 상태인 `ContractRecurringItem`과 승인된 `AdhocCharge`를 계약별로 그룹핑해 미리보기 제공
3. 경리가 확인 후 일괄 생성 → `Invoice`(초안 상태) + `InvoiceLineItem` 생성
4. 경리가 개별 또는 일괄로 "발행" 실행 → PDF 생성 + 고객 이메일 자동 발송 → 상태를 발송완료로 전환
5. 고객은 포털에 로그인해 본인 계약의 청구서 목록과 PDF를 조회

**중복 청구 방지**: 이미 청구서에 포함된 `ContractRecurringItem`/`AdhocCharge`는 다음 생성 미리보기에서 제외한다.

## 이메일/PDF 발송

- PDF는 서버에서 생성(청구서 템플릿 → PDF 렌더링)하고 `InvoicePdf`로 버전 관리해 저장.
- 이메일 발송은 실제 발송 로직을 인터페이스로 추상화한다(`InvoiceMailer` 등). 운영 환경에서는 실제 이메일 서비스 구현체를 사용하고, 테스트에서는 mock 구현체로 대체한다. 구체적인 이메일 발송 서비스(SMTP/서드파티) 선정은 구현 단계에서 결정한다.

## 테스트 전략

- 청구서 생성 로직(정액+건별 항목 병합, 중복 청구 방지, 금액 스냅샷)은 NestJS 서비스 단위 테스트로 검증.
- 권한 가드(역할별 접근 제어)는 각 역할에 대한 통합 테스트로 검증.
- 이메일 발송은 mock `InvoiceMailer`로 대체해 실제 외부 발송 없이 검증.

## 기술 스택

- Frontend: React (admin-web, portal-web 각각 별도 SPA)
- Backend: NestJS (단일 API 서비스)
- Database: PostgreSQL
